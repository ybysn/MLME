## 结论

有优化空间，而且很明显。你现在的流程是：

```text
你问我 → 我给一大段 OpenCode 指令 → OpenCode 执行 → 你截图反馈 → 我再修正
```

这个流程能推进，但缺点是：**OpenCode 被你当成“执行器”，没有充分利用它自己的 Plan / Build / Agents / CLI 能力**。
更优流程应该改成：

```text
我负责：架构边界 + 验收标准 + 风险控制
OpenCode Plan：读代码、给实施计划、不改代码
你确认：只确认影响面和方案
OpenCode Build：小步改代码
OpenCode Review/Run：自查 diff、跑 build、列风险
你验收：只看功能和回归
```

OpenCode 官方文档本身也建议先用 Plan mode 制定计划，再切回 Build mode 执行；Plan mode 是只分析、不直接改代码的模式。([OpenCode](https://opencode.ai/docs/?utm_source=chatgpt.com))

------

# 一、你现在流程的主要问题

| 问题                     | 结果                    | 优化方向                                          |
| ------------------------ | ----------------------- | ------------------------------------------------- |
| 指令太长，一次性塞太多   | OpenCode 容易改大、改偏 | 拆成 Plan 指令和 Build 指令                       |
| 我直接给最终实现方案     | 有时没基于当前真实代码  | 先让 OpenCode Plan 扫代码                         |
| 你主要靠截图反馈         | 定位慢                  | 让 OpenCode 输出“改动文件 + diff 摘要 + 验证结果” |
| Build 阶段缺 review      | 控制台红错容易漏        | 每次 Build 后追加 Review 指令                     |
| 功能和回归混在一起       | 容易修一个坏一个        | 每步固定“功能验收 + 核心闭环回归”                 |
| 没充分用 OpenCode agents | Plan/Build职责没分清    | 固定 Plan → Build → Review 三段式                 |

------

# 二、推荐新流程：三段式

## 1. Plan 阶段：只分析，不改代码

你先在 OpenCode 里切到 Plan mode。OpenCode 官方说明可以用 Tab 在 Plan/Build 等 primary agents 间切换，也可以配置/使用不同 agent。([OpenCode](https://opencode.ai/docs/agents/?utm_source=chatgpt.com))

发这种短指令：

```text
进入 Plan 模式。本阶段只读代码，不修改文件。

任务：分析如何实现【功能名称】。

要求：
1. 先阅读 AGENTS.md、docs/PRD.md、docs/architecture.md、docs/markdown_spec.md。
2. 找出相关文件和现有逻辑。
3. 明确是否影响：
   - Markdown 正文
   - 写作模式
   - 图片资产链路
   - 保存/自动保存
   - 导出 HTML/PDF
   - 窗口关闭守卫
4. 输出实施计划。
5. 不要修改代码。
```

这个阶段的输出你发给我，我帮你判断方案是否会破坏边界。

------

## 2. Build 阶段：小步执行

方案确认后，再切 Build mode。

```text
进入 Build 模式。按刚才 Plan 阶段确认的方案执行。

限制：
1. 只改 Plan 中列出的文件。
2. 不碰图片资产链路，除非本任务明确涉及图片。
3. 不碰窗口关闭守卫，除非本任务明确涉及关闭。
4. 不做无关重构。
5. 完成后运行 pnpm build。
6. 输出：
   - 改动文件
   - 核心逻辑
   - 已运行命令
   - 风险/未完成
```

这样 OpenCode 不会一上来乱改。

------

## 3. Review 阶段：让 OpenCode 自查

Build 完后，不要马上继续新功能。追加：

```text
进入 Review。不要修改代码，先审查刚才改动。

检查：
1. git diff --stat。
2. 是否改了无关文件。
3. 是否破坏 Markdown 正文保存。
4. 是否可能影响图片路径。
5. 是否可能影响写作模式生命周期。
6. 是否可能影响窗口关闭守卫。
7. 是否有控制台潜在错误。
8. 是否需要补测试样本。

输出：
- 问题列表
- 是否建议提交
- 需要我手动验证的点
```

这一步能显著降低“功能正常但控制台红错”的概率。

# 三、结合 OpenCode 功能，你应该这样用

| OpenCode 能力  | 你现在用法     | 推荐用法                                 |
| -------------- | -------------- | ---------------------------------------- |
| Plan agent     | 基本没充分用   | 每个中高风险任务必须先 Plan              |
| Build agent    | 直接执行大指令 | 只执行确认后的局部方案                   |
| Agent 切换     | 偶尔用         | 固定：Plan → Build → Review              |
| 自定义 agents  | 未充分使用     | 建议配置 `reviewer` / `test` agent       |
| `opencode run` | 少用           | 用于重复检查/批量审查                    |
| 多 session     | 未结构化       | 每个大功能单独 session，避免上下文污染   |
| 图片输入       | 已用截图反馈   | UI 问题继续用截图，代码问题优先日志/diff |

OpenCode CLI 支持 `opencode run "..."` 这种程序化调用，适合做重复性审查或生成总结，而不是每次都进 TUI 手动对话。([OpenCode](https://opencode.ai/docs/cli/?utm_source=chatgpt.com))

# 四、建议你配置 3 类 agent

OpenCode 支持配置 specialized agents，并且 agents 可以有不同权限、模型和工具访问范围。([OpenCode](https://opencode.ai/docs/agents/?utm_source=chatgpt.com))

## 1. Plan agent

职责：读代码、出方案、不改文件。

适合模型：推理强的模型。
权限：禁用 edit，限制 shell。

用途：

```text
这个功能该怎么接入？
会影响哪些模块？
有没有现有逻辑可复用？
```

## 2. Build agent

职责：按方案改代码。

适合模型：代码执行强、稳定。
权限：允许 edit / bash。

用途：

```text
只按确认方案改这几个文件。
```

## 3. Review agent

职责：审查 diff、找回归风险。

权限：只读 + bash。
用途：

```text
检查这次改动有没有污染图片链路/保存闭环。
```

这比一个 agent 从头干到尾稳很多。

# 八、提交策略也要固定

每个功能必须有 checkpoint：

```powershell
pnpm build
git status
git diff --stat
git add .
git commit -m "type: message"
```

建议 commit 类型：

| 类型        | 用法             |
| ----------- | ---------------- |
| `feat:`     | 新功能           |
| `fix:`      | bug 修复         |
| `refactor:` | 重构             |
| `style:`    | UI/CSS           |
| `chore:`    | 配置、checkpoint |
| `test:`     | 测试样本         |

图片链路、关闭守卫这类稳定点必须单独提交，不要和新功能混在一个 commit。

## 最终建议

你的最佳组合应该是：

```text
你：产品判断 + 最终验收
我：架构边界 + 风险控制 + 指令设计
OpenCode Plan：读取真实代码，制定方案

OpenCode Review：审查 diff 和回归风险

```

这样既能利用 OpenCode 的代码上下文能力，又不会让它在复杂链路里自由发挥。
