# MarkdownEditor 产品需求文档

## 0. 文档信息

| 字段 | 内容 |
|---|---|
| 产品名称 | MarkdownEditor |
| 产品定位 | 本地优先、轻量、稳定、类 Typora 的 Markdown 桌面编辑器 |
| 第一阶段目标 | 做出可长期维护的本地 Markdown 编辑器 MVP |
| 目标平台 | Windows / macOS / Linux |

## 1. 产品目标

### 1.1 核心目标

MarkdownEditor 的第一阶段目标不是复制 Typora 的全部功能，而是先完成一个稳定、可扩展、可测试的本地 Markdown 编辑器。

核心目标：

1. 支持本地 `.md` 文件的新建、打开、编辑、保存、另存为。
2. 支持类 Typora 的实时编辑体验，尽量减少源码和预览之间的割裂。
3. 支持常用 Markdown 语法：标题、列表、引用、代码块、表格、图片、链接、数学公式、Mermaid 图表。
4. 支持本地图片拖拽插入，并自动复制到当前文档对应的 `.assets` 目录。
5. 支持大纲、字数统计、主题切换、最近文件、基础导出。
6. 保证 Markdown 加载、编辑、保存后结构稳定，不出现明显语义破坏。

### 1.2 产品判断

第一版应优先做成：

> 稳定的本地 Markdown 编辑器 + 基础所见即所得体验

而不是：

> 一开始就完全复刻 Typora 的所有视觉和交互细节

原因：完全 Typora 式 WYSIWYG 编辑的难点集中在光标、选区、输入法、撤销栈、Markdown AST 序列化、表格编辑和图片路径处理。如果第一版直接硬做全量能力，会导致工程不可控。

### 3.2 第一阶段明确不做

| 不做事项 | 原因 |
|---|---|
| 云同步 | 会引入账户、冲突、网络、安全复杂度 |
| 多人协作 | 与本地优先编辑器目标冲突 |
| 插件市场 | 第一版没有必要增加扩展系统复杂度 |
| 完整主题市场 | 先支持 CSS 变量和明暗主题 |
| 完整博客发布 | 与编辑器核心闭环无关 |
| 大模型深度接管编辑器 | AI 功能后置，避免污染核心编辑链路 |
| 自研 contenteditable 编辑器内核 | 风险极高，不应重复造轮子 |

## 4. 核心使用流程

### 4.1 新建文档

1. 用户打开应用。
2. 点击“新建文档”。
3. 编辑器进入空白状态。
4. 用户输入内容。
5. 用户点击保存或 Ctrl+S。
6. 如果尚未选择路径，弹出保存路径选择框。
7. 保存为 `.md` 文件。

验收标准：

- 未保存文档关闭时必须提示。
- 新建文档必须有脏状态标识。
- 保存成功后脏状态清除。

### 4.2 打开文档

1. 用户点击“打开文件”。
2. 选择本地 `.md` 文件。
3. Rust 文件层读取内容。
4. 前端编辑器加载 Markdown。
5. 最近文件列表更新。

验收标准：

- 文件不存在时给出明确错误。
- 非 UTF-8 文件需要提示编码问题，不静默损坏内容。
- 打开大文件时如超过阈值，需要提示可能降级。

### 4.3 图片拖拽

1. 用户将图片拖入编辑器。
2. 系统判断当前文档是否已保存。
3. 如果未保存，提示先保存文档。
4. 如果已保存，复制图片到 `当前文档名.assets/`。
5. 插入相对路径 Markdown 图片语法。

示例：

```md
![image-20260522-153000](./note.assets/image-20260522-153000.png)
```

验收标准：

- 支持 png / jpg / jpeg / gif / webp / svg。
- 重名文件自动追加序号或 hash。
- 插入路径必须为相对路径。

### 4.4 导出 HTML

1. 用户点击“导出 HTML”。
2. 系统读取当前编辑器内容。
3. 转换为 HTML。
4. 注入主题 CSS。
5. 处理本地图片路径。
6. 输出 `.html` 文件。

验收标准：

- 标题、列表、代码块、表格、图片、公式样式正常。
- 相对图片路径在导出目录中可访问。

## 6. 功能验收标准

### 6.1 MVP 验收

MVP 通过标准：

1. 能在 Windows / macOS / Linux 至少一个平台稳定启动。
2. 能新建、打开、保存、另存为 `.md` 文件。
3. 能正常编辑标题、列表、引用、代码块、表格、图片、链接。
4. 能拖拽图片到文档并生成相对路径。
5. 能显示右侧大纲。
6. 能切换明暗主题。
7. 能导出 HTML。
8. 核心 Markdown 样本 round-trip 测试通过。
9. 无明显数据丢失问题。
10. Ctrl+S 保存、关闭未保存提示正常。

### 6.2 Round-trip 验收

每个测试样本必须经过：

```text
原始 Markdown
→ 编辑器加载
→ 编辑器内部文档模型
→ 序列化 Markdown
→ 再次加载
→ 对比结构一致性
```

第一阶段不强制字符级完全一致，但必须保证语义结构一致。

## 7. 性能要求

| 场景 | 要求 |
|---|---|
| 打开 100KB 文档 | 1 秒内可编辑 |
| 打开 1MB 文档 | 3 秒内可编辑 |
| 打开 5MB 文档 | 可提示性能风险 |
| 自动保存 | 不阻塞输入 |
| 大纲刷新 | debounce，不每次输入全量重算 |
| 图片复制 | 大图需要异步处理 |

---

## 8. 安全与可靠性要求

| 要求 | 说明 |
|---|---|
| 文件写入前备份 | 重要保存操作可写临时文件再替换 |
| 不静默覆盖 | 另存为覆盖文件前提示 |
| 错误可见 | 文件读写失败必须显示原因 |
| 路径安全 | 防止越权访问非用户选择路径 |
| 本地优先 | 默认不上传任何内容 |
| AI 功能隔离 | AI 不应默认读取用户全部文件夹 |

## 10. 第一阶段结论

第一阶段只追求一个目标：

> 文件闭环稳定、Markdown 结构稳定、编辑体验可用。

只要这三点没有稳定，任何 AI、云同步、主题市场、插件系统都不应提前开发。
