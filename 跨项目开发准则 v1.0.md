# 跨项目开发准则 v1.0

## 1. 总原则

| 准则       | 要求                                                       |
| ---------- | ---------------------------------------------------------- |
| 边界优先   | 新增功能前，先判断属于哪个模块                             |
| 单向依赖   | UI → service → repository → adapter，不允许反向调用        |
| 成熟优先   | 有成熟框架、库、组件时，优先采用成熟方案                   |
| 业务自研   | 只在项目特有业务逻辑处自研                                 |
| 小步交付   | 每次只改一组相关问题，不混杂多个方向                       |
| 补丁最小化 | 只发改动文件，不发全量项目                                 |
| 可回归     | 每次修改必须能说明如何验证                                 |
| 可追踪     | 关键任务、流水线、交易、订单、执行结果必须有日志或状态记录 |
| 可替换     | 数据源、模型、策略、执行端、UI组件尽量用 adapter 隔离      |
| 可删除     | 新逻辑稳定后，旧逻辑必须删除或标记废弃                     |

------

# 2. 项目结构准则

## 2.1 按业务能力拆模块，不按技术类型堆目录

不推荐：

```text
src/
  components/
  utils/
  services/
  api/
  pages/
```

推荐：

```text
src/
  app/
  account/
  market/
  strategy/
  backtest/
  tracking/
  editor/
  assets/
  jobs/
  shared/
```

原因：
按技术分层早期清晰，后期必然膨胀。所有东西都会塞进 `utils`、`components`、`services`，最后边界失控。

------

## 2.2 每个模块内部固定分层

推荐结构：

```text
module_name/
  api/
  service/
  repository/
  adapter/
  types/
  components/
  hooks/
  tests/
  README.md
```

| 层         | 职责                             |
| ---------- | -------------------------------- |
| api        | 对外接口、路由、请求响应         |
| service    | 业务规则、流程编排               |
| repository | 数据读写                         |
| adapter    | 第三方系统、外部库、平台能力封装 |
| types      | 类型、枚举、schema               |
| components | UI 展示组件                      |
| hooks      | 前端状态和交互逻辑               |
| tests      | 单元测试、集成测试、回归测试     |
| README.md  | 模块边界、数据流、禁止事项       |

------

# 3. 依赖准则

## 3.1 允许的依赖方向

```text
UI / Page
  ↓
Hook / Controller
  ↓
Service
  ↓
Repository
  ↓
Adapter / DB / FileSystem / Third-party API
```

## 3.2 禁止事项

| 禁止行为                         | 原因                 |
| -------------------------------- | -------------------- |
| UI 直接读写数据库/文件           | 后期无法复用，难测试 |
| repository 写业务规则            | 数据层污染           |
| service 直接操作 DOM             | 业务层和表现层耦合   |
| adapter 反向调用 service         | 外部接口层反客为主   |
| shared 依赖业务模块              | 公共层失控           |
| 一个模块绕过另一个模块的公开接口 | 破坏封装             |

------

# 4. 文件规模准则

| 指标                            | 处理规则                  |
| ------------------------------- | ------------------------- |
| 单文件 > 300 行                 | 检查是否需要拆分          |
| 单文件 > 500 行                 | 原则上必须拆              |
| 单函数 > 60 行                  | 检查职责是否过多          |
| 单函数 > 100 行                 | 原则上必须拆              |
| 单文件 import > 12 个           | 检查模块职责              |
| 同类逻辑出现第 3 次             | 抽象成 service/helper     |
| 一个 bug 牵涉 4 个以上文件      | 说明边界需要重构          |
| 一个模块被 5 个以上模块直接依赖 | 考虑抽成公共能力或 facade |

------

# 5. shared / utils 使用准则

## 5.1 shared 只能放真正通用能力

允许：

```text
shared/
  logger/
  result/
  date/
  path/
  error/
  http/
  ui/
```

谨慎：

```text
shared/
  utils.ts
  helper.ts
  common.ts
```

## 5.2 判断是否能进入 shared

| 问题                      | 判断                  |
| ------------------------- | --------------------- |
| 是否被 3 个以上模块复用？ | 否，则不要放 shared   |
| 是否包含业务含义？        | 是，则放回业务模块    |
| 是否依赖业务类型？        | 是，则不能进 shared   |
| 是否只是为了“方便引用”？  | 是，则不应该放 shared |

# 6. 新功能开发准则

每个新功能必须先回答 6 个问题：

| 问题               | 目的           |
| ------------------ | -------------- |
| 属于哪个业务模块？ | 防止乱放       |
| 是否已有相似逻辑？ | 防止重复造轮子 |
| 是否能复用成熟库？ | 降低维护成本   |
| 对外接口是什么？   | 明确边界       |
| 影响哪些旧功能？   | 控制风险       |
| 如何验证成功？     | 保证可回归     |

默认流程：

```text
需求确认
  ↓
查找现有逻辑
  ↓
设计最小改动方案
  ↓
实现
  ↓
本地验证
  ↓
打补丁包
  ↓
说明修改文件和验证命令
```

------

# 7. 修改旧功能准则

旧功能修改不能只看“能不能跑”，必须看：

| 检查项           | 要求               |
| ---------------- | ------------------ |
| 是否破坏旧接口   | 不允许无说明破坏   |
| 是否引入重复逻辑 | 不允许             |
| 是否保留废弃代码 | 稳定后必须清理     |
| 是否影响数据口径 | 必须说明           |
| 是否影响前端展示 | 必须同步检查       |
| 是否影响任务链路 | 必须检查日志和状态 |
| 是否有回滚方案   | 关键模块必须有     |

# 8. 数据与任务链路准则

适用于量化、订单、自动化、编辑器文件处理等所有项目。

## 8.1 关键链路必须可追踪

| 链路         | 必须记录                               |
| ------------ | -------------------------------------- |
| 数据同步     | 日期、数据源、数量、失败原因           |
| 策略计算     | 参数、版本、输入日期、输出数量         |
| 回测         | run_id、配置、收益曲线、归因、错误     |
| 自动发货     | 订单号、模板、卡密、发送状态、失败原因 |
| 浏览器自动化 | session、风控状态、截图、错误节点      |
| 文件处理     | 原文件、保存路径、资源引用、失败原因   |
| 导出功能     | 输入文档、导出格式、输出路径、异常     |

## 8.2 任务状态必须明确

禁止只有：

```text
success / failed
```

推荐：

```text
queued
running
success
partial_success
failed
cancelled
manual_required
expired
skipped
```

# 9. 前端开发准则

## 9.1 页面不要堆功能

每个复杂页面必须分：

| 层级     | 作用                       |
| -------- | -------------------------- |
| 普通模式 | 用户主路径，少按钮、少参数 |
| 高级模式 | 专业配置、调参、诊断       |
| 诊断区   | 错误、日志、数据质量       |
| 历史区   | 任务、快照、记录           |

## 9.2 前端文件拆分规则

推荐：

```text
frontend/
  pages/
    dashboard/
      index.tsx
      components/
      hooks/
      service.ts
      types.ts
  shared/
    components/
    hooks/
    utils/
```

禁止一个页面文件同时承担：

```text
API请求
状态管理
表格渲染
图表配置
弹窗逻辑
业务判断
错误处理
```

# 10. 后端开发准则

## 10.1 后端必须遵守分层

```text
routes / controllers
  ↓
services
  ↓
repositories
  ↓
db / external adapters
```

| 层         | 禁止                     |
| ---------- | ------------------------ |
| routes     | 禁止直接写 SQL           |
| service    | 禁止直接拼 HTTP 响应格式 |
| repository | 禁止业务判断             |
| db         | 禁止知道上层业务语义     |
| adapter    | 禁止污染业务模型         |

------

# 11. 第三方依赖准则

## 11.1 成熟方案优先

| 场景     | 优先选择                                           |
| -------- | -------------------------------------------------- |
| 图表     | ECharts / Recharts / TradingView 类成熟方案        |
| Markdown | Milkdown / CodeMirror / markdown-it / unified 生态 |
| 表格     | TanStack Table / AG Grid 等                        |
| 状态管理 | Zustand / Redux Toolkit / Vue Pinia                |
| 后端 API | FastAPI / Django / NestJS                          |
| 任务队列 | Celery / RQ / Dramatiq                             |
| 数据校验 | Pydantic / Zod                                     |
| ORM/SQL  | SQLAlchemy / Prisma / Drizzle                      |
| 自动化   | Playwright                                         |
| 日志     | structlog / loguru / 标准 logging 封装             |

## 11.2 什么时候自研

只在以下情况自研：

| 条件                     | 是否自研   |
| ------------------------ | ---------- |
| 项目核心竞争力           | 可以自研   |
| 第三方方案明显不满足业务 | 可以自研   |
| 只是普通基础能力         | 不建议自研 |
| 只是觉得“自己写更快”     | 不建议自研 |
| 后续维护成本不可控       | 不建议自研 |

以后默认交付格式：

```text
本次修改目标：
- xxx

- xxx
- xxx

- xxx

- xxx

- xxx
```

```text

不发全量项目

```

# 13. 技术债管理准则

每次开发结束必须做一次收敛检查：

| 检查项           | 动作            |
| ---------------- | --------------- |
| 是否新增重复逻辑 | 合并            |
| 是否留下临时代码 | 删除或标记 TODO |
| 是否新增无用参数 | 删除            |
| 是否新增无用接口 | 删除            |
| 是否有旧逻辑并存 | 明确迁移或废弃  |
| 是否有测试缺口   | 加入回归清单    |
| 是否有文档缺口   | 更新模块说明    |

# 14. 版本迭代规则

这份准则后续按版本升级。

| 版本 | 触发条件                             |
| ---- | ------------------------------------ |
| v1.0 | 当前通用基线                         |
| v1.1 | 某项目出现重复问题，需要加入新规则   |
| v1.2 | 某类架构反复失控，需要强化边界       |
| v2.0 | 多项目经验成熟后，重构成正式工程手册 |

每次新增规则必须说明：

```text
新增规则：
为什么新增：
适用范围：
不适用范围：
执行方式：
```

------

# 15. 最重要的硬规则

以后所有项目默认遵守下面 10 条：

| 序号 | 硬规则                           |
| ---- | -------------------------------- |
| 1    | 新增功能前先查是否已有相似逻辑   |
| 2    | 不重复造轮子，成熟方案优先       |
| 3    | 业务逻辑不得写进 UI 组件         |
| 4    | routes 不直接操作数据库          |
| 5    | service 不直接依赖具体第三方平台 |
| 6    | 公共层不能变垃圾桶               |
| 7    | 一个链路只能有一个主入口         |
| 8    | 修改必须附验证方式               |
| 9    | 稳定后必须删除旧逻辑             |
| 10   | 每轮开发后必须做技术债收敛       |
